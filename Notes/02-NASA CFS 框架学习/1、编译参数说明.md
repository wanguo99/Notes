NASA **cFS**（Core Flight System）的编译主要通过顶层 **Makefile**（从 `cfe/cmake/Makefile.sample` 复制而来）驱动，底层使用 **CMake**。编译过程分为几个阶段：**prep**（配置/生成构建文件）、**build**（实际编译）、**install**（安装可执行文件和启动脚本）。

cFS 的编译参数主要分为两大类：

1. **Makefile 顶层参数**（最常用，通过 `make XXX=yyy` 传递）
2. **CMake 配置参数**（在 `prep` 阶段传递，或通过 `sample_defs/` 中的配置文件修改）

下面是常见且实用的参数及其功能说明（基于 cFS 主线 Draco/Caelum 等版本，推荐查看 `cfe/cmake/README.md` 获取最新细节）。

### 1. Makefile 顶层主要参数（直接在 make 命令中使用）

| 参数                  | 示例                                                                  | 功能说明                                                                                                                                                                                                                          |
| ------------------- | ------------------------------------------------------------------- | ----------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------- |
| **SIMULATION**      | `SIMULATION=native`<br>`SIMULATION=arm-cortexa8_neon-linux-gnueabi` | **最重要参数**。指定目标架构/模拟模式。<br>- `native`：使用宿主机原生编译器（gcc），适合在 PC/Linux 上快速测试运行，不进行交叉编译。<br>- 其他值：覆盖 `targets.cmake` 中的架构设置，用于 Processor-In-the-Loop (PIL) 或真实目标交叉编译（如你的 Buildroot aarch64）。<br>如果不设置，则使用 `targets.cmake` 中定义的系统类型。 |
| **O**               | `O=build`<br>`O=build-sim`                                          | 指定输出构建目录（可相对或绝对路径）。<br>默认是 `build`。不同构建（如 native 和 cross）建议使用不同目录，避免冲突。                                                                                                                                                       |
| **CFE_EDS_ENABLED** | `CFE_EDS_ENABLED=TRUE`                                              | 启用 **Electronic Data Sheets (EDS)** 支持（较新功能）。允许使用 EDS 定义命令/遥测格式，提升与地面系统的互操作性。示例：`make SIMULATION=native CFE_EDS_ENABLED=TRUE prep`。                                                                                           |
| （无参数）               | `make prep`<br>`make`<br>`make install`                             | - `prep`：运行 CMake 配置，生成构建文件（必须先执行）。<br>- `make`（无目标）：执行编译。<br>- `make install`：安装到 `build/exe/cpu1/` 等目录（包含 core-cpu1、可加载 .so 应用、启动脚本）。<br>快捷方式：`make SIMULATION=native install` 可一次性完成 prep + build + install。               |
| **BUILDTYPE** 或类似   | （部分项目支持）                                                            | 部分扩展构建中可设置 Debug/Release，但 cFS 主要通过 CMake 的 `CMAKE_BUILD_TYPE` 控制（见下文）。                                                                                                                                                       |

**常用完整命令示例**：
- 本地快速测试：`make SIMULATION=native prep && make && make install`
- 交叉编译（Buildroot 风格）：先在 `sample_defs/toolchain-xxx.cmake` 中定义工具链，然后 `make prep`（或带 SIMULATION 参数）。

### 2. 配置层参数（影响 prep 阶段的行为）

这些参数主要通过以下方式设置：
- 修改 `sample_defs/` 目录下的文件（推荐方式，`targets.cmake`、`default_osconfig.cmake`、`cfe_mission_cfg.h` 等）。
- 或在 `prep` 时通过环境变量 / CMake cache 传递（`-Dxxx=yyy`）。

#### (1) **targets.cmake**（核心目标配置）
- 定义每个 CPU（cpu1、cpu2...）的信息：
  - `TGT1_NAME`、`TGT1_SYSTEM`（架构，如 `generic-linux`）
  - `TGT1_PLATFORM`（平台配置）
  - `TGT1_APPLIST`（要编译的应用列表，例如 `sample_app ci_lab to_lab`）
  - `TGT1_FILELIST`（要复制的文件，如启动脚本 `cfe_es_startup.scr`）
- 功能：决定构建哪些应用、针对哪些 CPU、如何复制启动脚本等。

#### (2) **OSAL 配置**（`default_osconfig.cmake` 或生成的 `osconfig.h`）
- `OSAL_CONFIG_DEBUG_PERMISSIVE_MODE = TRUE`：**非常实用**。在普通 Linux（非 root）下运行时，放松消息队列深度、任务优先级等限制，避免权限或资源问题导致启动失败。
- 其他 OSAL 参数：队列大小、任务栈大小、文件系统映射等。

#### (3) **cFE / cFS 配置参数**（`cfe_platform_cfg.h`、`cfe_mission_cfg.h` 等）
这些是 **编译时常量**，影响内存占用、功能启用、性能等。主要类别包括：

- **Executive Services (ES)**：最大应用数、任务数、堆大小、性能 ID 等。
- **Software Bus (SB)**：管道深度、消息大小、订阅限制。
- **Event Services (EVS)**：最大事件消息长度、日志大小。
- **Table Services (TBL)**：
  - `CFE_PLATFORM_TBL_BUF_MEMORY_BYTES`：表缓冲池大小
  - `CFE_PLATFORM_TBL_MAX_NUM_TABLES`：最大表数量
  - 双缓冲 vs 单缓冲选项等。
- **Time Services**：服务器/客户端模式、时间源（内部/外部/GPS）、飞轮模式、任务优先级/栈大小。
- **File Services (FS)**、**Critical Data Store (CDS)** 等。

这些参数大多定义在 `cfe/fsw/platform_inc/` 或 `mission_inc/` 下，项目通常通过 `sample_defs/` 中的 wrapper 文件覆盖。

- **Mission vs Platform 配置**：
  - **Mission** 配置：全任务通用（如时间 epoch、最大消息大小）。
  - **Platform** 配置：特定 CPU/处理器专用（如最大应用数、时间服务器设置）。

#### (4) **其他 CMake 传递参数**
- 在 `prep` 时可直接加：`make ... -D<CMAKE_VAR>=value`
- 常见：`CMAKE_BUILD_TYPE=Debug/Release`（控制优化/调试符号）、工具链文件路径等。

### 3. Buildroot 集成中的注意事项（结合你之前的自定义 package）
- 在 `CFS_CONFIGURE_CMDS` 中调用 `make ... SIMULATION=buildroot O=build prep`（或类似），并通过 `toolchain-buildroot.cmake` 传递交叉工具链（`TARGET_CC`、`TARGET_CFLAGS` 等）。
- 推荐在 `sample_defs/default_osconfig.cmake` 中启用 `OSAL_CONFIG_DEBUG_PERMISSIVE_MODE=TRUE`（非 root 运行时特别有用）。
- 如果想动态调整应用列表或配置，修改 `targets.cmake` 后重新 `make cfs-rebuild`。
- 启用特定功能（如 EDS）：在 prep 命令中添加 `CFE_EDS_ENABLED=TRUE`。

### 4. 推荐实践流程
1. 复制 `Makefile.sample` 和 `sample_defs/`。
2. 修改 `sample_defs/targets.cmake`（添加你的应用、平台）。
3. 修改 `default_osconfig.cmake`（permissive mode 等）。
4. `make SIMULATION=xxx prep`（生成 CMake 配置）。
5. `make`（编译）。
6. `make install`（得到 `build/exe/cpu1/` 目录，可直接复制到目标板 `/opt/cfs/`）。

**调试建议**：
- 第一次构建失败时，先检查 `build/` 目录下的 CMakeCache.txt 或日志。
- 常见问题：队列深度不够（调高 `/proc/sys/fs/mqueue/msg_max` 或用 permissive mode）、工具链不匹配、子模块未更新。
- 想查看所有可用 CMake 选项：运行 prep 后查看生成的 CMakeCache.txt，或直接阅读 cFE/cFS 的 CMakeLists.txt。

如果你正在 Buildroot 中集成，需要针对特定参数的修改示例（如如何在 cfs.mk 中自动设置某个 CFE_xxx 参数、或自定义 targets.cmake），或者遇到具体报错，请提供你的当前 `sample_defs/` 配置片段或错误信息，我可以给出更精确的调整建议。cFS 的配置非常灵活，主要靠这些参数实现“一次配置、多平台复用”。
